# VLM 闭环泊车测试框架 — 完整技术总结

## 1. 系统概述

为语言引导泊车探索模型（ExpVLA）在 CARLA 仿真环境中构建的闭环自动测试框架。系统在 ROS Noetic 下运行，通过循环调用 VLM 服务器，控制车辆执行探索或泊车动作，最终输出 SPL/APE/AOE 等性能指标。

## 3. 坐标系设计

```
body (ego车体) ──→ camera_init (FastLIO原点) ──TF──→ map (CARLA世界)
```

| 坐标系 | 来源 | 用途 |
|---|---|---|
| `map` | CARLA 世界坐标 | `set_transform` 发布目标、`gt_trajectory.csv` 内容 |
| `camera_init` | FastLIO 原点（每任务重置） | 里程计、泊车位、Hybrid A* 路径、VLM 输入位姿 |
| `body` | 自车车体 | VLM 输出轨迹 |

- `camera_init → map` TF 由 `map_to_init_tf.py`（内嵌于 FastLIO launch）持续发布
- 所有内部位姿跟踪在 `camera_init` 系下进行，仅在 `publish set_transform` 时转换为 `map`
- VLM 输入的 `curr_pose` / `history_poses` 均为 `camera_init` 系，yaw 单位为**度（degrees）**

## 4. 任务数据格式

每个任务文件夹结构：

```
town04/2026-05-15-10-49/
├── gt_trajectory.csv       # 地面真值轨迹
├── instruct.txt            # 语言指令（UTF-8，单行）
└── scenario_layout.yaml    # 场景车辆布局（可选）
```

`gt_trajectory.csv` 列格式：

```
timestamp, x, y, z, roll, pitch, yaw
```

- 坐标系：CARLA `map` 系
- yaw 单位：**弧度（radians）**
- 第 0 行为自车起始位置，最后一行为目标泊车位置

## 5. VLM 服务器接口

**Endpoint:** `POST http://localhost:9999/predict`

**请求 payload（JSON）：**

```json
{
  "instruction": "Park in the nearest available slot on the left",
  "image_front_b64": "<base64 PNG>",
  "image_rear_b64":  "<base64 PNG>",
  "image_left_b64":  "<base64 PNG>",
  "image_right_b64": "<base64 PNG>",
  "history_poses": [{"x": 1.0, "y": 2.0, "yaw": 45.0}, ...],
  "curr_pose":     {"x": 3.0, "y": 4.0, "yaw": 90.0},
  "parking_slots": [{"id": 1, "x": 5.0, "y": 6.0}, ...]
}
```

- 图片：**客户端不做 resize**，服务端 `PrismaticImageProcessor` 处理所有预处理
- 位姿：`camera_init` 系，yaw 单位度
- `history_poses`：不含最新帧（最新帧作为 `curr_pose`）；首次调用时为 `[]`

**响应：**

```json
{
  "decision_id": 0,
  "trajectory": [{"x": 1.0, "y": 0.5, "cos_yaw": 0.98, "sin_yaw": 0.2}, ...]
}
```

- `decision_id ≤ 0`：继续探索，执行 `trajectory`（body 系，8个点）
- `decision_id > 0`：泊入该 id 的车位，触发 Hybrid A*

## 6. 模块架构（`closetest_main.py`）

### 6.1 KeyframeTracker

- 订阅：`/Odometry_fastlio` (`nav_msgs/Odometry`)
- 定时器：2 Hz（`rospy.Timer(Duration(0.5))`）
- 关键帧条件：距离 > `kf_dist_thresh`（默认 0.1m）**或** yaw 变化 > `kf_yaw_thresh`（默认 5°）
- 存储格式：`{x, y, yaw_deg}` 列表，`camera_init` 系
- `get_history()` 返回**除最新帧以外**的所有关键帧（最新帧作为 `curr_pose` 传给 VLM）
- `get_curr_odom()` 返回最新 Odometry 消息
- `get_trajectory_xy()` 返回所有关键帧的 `(x, y)` 列表，用于 SPL 计算

### 6.2 ParkingSpaceManager

- 订阅：`/parking_map/confirmed_spaces` (`ParkingSpaceArray`，`camera_init` 系)
- 以 `id` 为键累积所有见过的车位（重复检测时更新）
- `get_all_slots()` → `[{id, x, y}]`，用于 VLM 输入
- `get_space_by_id(id)` → 完整 `ParkingSpace` 消息，用于 Hybrid A* 目标

### 6.3 ImageBuffer

- 订阅：`/carla/ego_vehicle/rgb_front/image`、`rgb_rear`、`rgb_left`、`rgb_right`
- 将每帧转为 base64 PNG（BGR→RGB 转换），`threading.Lock` 保护
- **不做 resize**

### 6.4 VLMClient

- `POST /predict`，timeout=120s
- 返回 `(decision_id: int, trajectory: list[dict])`

### 6.5 NavigationExecutor

- 发布：`/carla/ego_vehicle/ackermann_cmd` (`ackermann_msgs/AckermannDrive`) — 探索阶段
- 发布：`/carla/ego_vehicle/control/set_transform` (`geometry_msgs/Pose`，`map` 系) — 泊车阶段
- 发布：`/vlm_trajectory_marker` (`visualization_msgs/Marker`，红色，`camera_init` 系)
- TF：`tf2_ros.Buffer` + `TransformListener`（`camera_init → map`）

**explore_navigate(trajectory_body, curr_odom, get_curr_odom)：**

1. 取 VLM 轨迹第 4 个路点（index 3，不足时取最后一个），**丢弃 VLM 的 yaw 输出**
2. 将目标点从 body 系转换到 `camera_init` 系（XY 平移旋转，不使用 VLM yaw）：
   ```
   tx = ox + bx*cos(oyaw) - by*sin(oyaw)
   ty = oy + bx*sin(oyaw) + by*cos(oyaw)
   ```
3. 将全部 8 个 VLM body 路点转换到 `camera_init` 系，发布红色轨迹 Marker（只显示最新一条）
4. 进入 Pure Pursuit Ackermann 控制循环（10 Hz）：
   - 实时读取最新里程计 → 计算与目标距离
   - 距离 < `arrive_thresh` → 发布 `speed=0` 停车，返回（触发 VLM 重新调用）
   - 否则：Pure Pursuit 计算转向角，以固定 `target_speed` 发布 `AckermannDrive`
5. Pure Pursuit 公式（Coulter 1992）：
   ```
   α = atan2(dy, dx) - cyaw      # 朝向误差
   δ = atan(2·L·sin(α) / ld)     # 转向角
   ```
   其中 `ld = hypot(tx-cx, ty-cy)`，`L = wheelbase`

> **注意**：VLM 推理期间（~5–10s）车辆自然停在目标点附近；速度 PID 由 `carla_ackermann_control` 内部实现，无需手写。

**park_navigate(path_msg)：**

- 先发布 `AckermannDrive(speed=0)` + sleep 0.5s，静默 Ackermann 控制器（防止与 set_transform 冲突）
- 跟随 Hybrid A* 路径（`nav_msgs/Path`，`camera_init` 系）
- 逐点 TF `camera_init→map` → publish Pose at `set_transform_rate` Hz

**RViz 可视化：**

- Topic：`/vlm_trajectory_marker`
- 类型：`Marker`，`LINE_STRIP`，红色（r=1, g=0, b=0），坐标系：`camera_init`
- **固定 `marker.id=0`**：每次 VLM 探索步骤发布新轨迹时替换旧的，RViz 中始终只显示最新一条

### 6.6 HybridAStarInterface

- 发布：`/parkman/planning/input/start`、`/parkman/planning/input/goal`（`PoseStamped`，`camera_init`）
- 订阅：`/parkman/planning/output/trajectory` (`nav_msgs/Path`)
- `request_path(curr_odom, goal_space)`：发布 3 次（确保收到），等待路径最多 `hybrid_astar_timeout` 秒

> **注意**：Hybrid A* 订阅 `/map` 话题，但 FastLIO 发布 `/local_map`。`setup_closetest_main.sh` 中通过 `rosrun topic_tools relay /local_map /map` 桥接。

### 6.7 CollisionMonitor

- 订阅：`/carla/ego_vehicle/collision` (`carla_msgs/CarlaCollisionEvent`)
- 碰撞后设置 flag，`had_collision()` 返回 bool

### 6.8 PerformanceEvaluator

结果目录：`carla_close_test/performance_result/{task_folder_name}/closetest_result_{timestamp}.txt`

**计算指标：**

| 指标 | 计算方法 |
|---|---|
| APE | `hypot(final_x - gt_final_x, final_y - gt_final_y)`（map 系，米） |
| correct_space | `APE < correct_space_thresh`（默认 3.0m） |
| AOE | `yaw_diff = abs(final_yaw_deg - gt_yaw_deg) % 360`；若 >180 取补；`aoe = min(yaw_diff, abs(180-yaw_diff))`（消除头朝里/朝外歧义） |
| Success (SR) | `not collision AND correct_space AND parking_attempted` |
| SPL | `gt_len / max(act_len, gt_len)` if success else 0 |

### 6.9 ClosedLoopTestNode（主编排器）

**主循环 run()：**

1. 等待传感器就绪（odom + 4路图像 + ≥1个泊车位），超时 60s
2. VLM 循环（最多 `max_vlm_steps` 次）：
   - 检查碰撞 → abort（记录失败指标，shutdown）
   - 检查任务超时（默认 600s）→ abort
   - 快照当前状态 → 调用 VLM
   - `decision_id > 0`：Hybrid A* 规划 → 跟踪执行 → `_finish()`
   - `decision_id ≤ 0`：`explore_navigate()` → 继续循环
3. 耗尽步数 → `_finish(timeout=True)`
4. `_finish()` → 评估指标 → 保存 txt → `rospy.signal_shutdown()`（解除 shell 阻塞）

## 7. 可配置 ROS 参数

所有参数均为 `closetest_main` 节点的私有参数（`~` 前缀）：

| 参数 | 默认值 | 说明 |
|---|---|---|
| `~task_dir` | 必填 | 任务文件夹路径 |
| `~vlm_server` | `http://localhost:9999` | VLM FastAPI 服务地址 |
| `~set_transform_rate` | `2.0` | 泊车阶段 set_transform 发布频率（Hz） |
| `~arrive_thresh` | `2.0` | 到达目标点触发 VLM 重调用的距离阈值（m） |
| `~target_speed` | `3.0` | 探索阶段巡航速度（m/s），由 Ackermann 内部 PID 跟踪 |
| `~wheelbase` | `2.875` | 车辆轴距（m），Tesla Model 3 |
| `~max_steer_angle` | `0.698` | Pure Pursuit 转向角限幅（rad，约 40°） |
| `~control_rate` | `10.0` | Ackermann 控制循环频率（Hz） |
| `~max_vlm_steps` | `50` | VLM 循环最大迭代次数 |
| `~kf_dist_thresh` | `0.1` | 关键帧距离阈值（m） |
| `~kf_yaw_thresh` | `5.0` | 关键帧 yaw 变化阈值（度） |
| `~correct_space_thresh` | `3.0` | 正确车位的 APE 阈值（m） |
| `~hybrid_astar_timeout` | `30.0` | 等待 Hybrid A* 路径的超时（s） |
| `~task_timeout` | `600.0` | 单任务最大允许时间（s），超时视为失败 |

## 8. Shell 脚本

### 8.1 `tools/setup_closetest_init_ego.sh`

- 参数 `$1`：TASK_DIR
- 读取 `gt_trajectory.csv` 第 0 行，5次 1Hz 发布 `set_transform` 将自车传送至起始位置
- **必须在 FastLIO 启动之前调用**（防止 LiDAR 跳变破坏里程计）

### 8.2 `tools/setup_closetest_main.sh`（单次任务测试）

- 参数：`$1` TASK_DIR，`$2` VLM_SERVER（默认 `http://localhost:9999`）
- 启动顺序：
  1. `rosrun topic_tools relay /local_map /map &`（捕获 PID）
  2. `roslaunch hybrid_a_star run_hybrid_a_star.launch &`（捕获 PID）
  3. `sleep 3`（等待 Hybrid A* 初始化）
  4. `rosrun carla_close_test closetest_main.py`（**阻塞，直到节点退出**）
  5. kill relay + hastar PIDs

### 8.3 `tools/setup_closetest_batch.sh`（批量测试）

- 参数：`$1` TOWN_DIR（含多个任务子文件夹的目录），`$2` VLM_SERVER
- 对每个含 `gt_trajectory.csv` 的子文件夹执行：

```
1. roslaunch spawn_scenario.launch yaml_path:=scenario_layout.yaml &  (sleep 8)
2. setup_closetest_init_ego.sh $TASK_DIR   ← 先初始化自车位置！
   sleep 3
3. setup_vps_map_dev_in_fastlio.sh &       ← 再启动 FastLIO（camera_init 干净重置）
   sleep 12
4. setup_closetest_main.sh $TASK_DIR $VLM_SERVER   ← 阻塞直到任务完成
5. kill FastLIO/BEV/scenario + pkill 残余进程
   sleep 3
```

- 任务完成信号：`closetest_main.py` 调用 `rospy.signal_shutdown()` → `rosrun` 退出 → shell 返回
- 批量结束后调用 `aggregate_results.py` 生成汇总报告

> **关键顺序约束**：`init_ego` 必须在 FastLIO 启动前完成。若 FastLIO 运行中传送自车位置，LiDAR 数据产生跳变，破坏 FastLIO 里程计积分。

### 8.4 `carla_close_test/scripts/aggregate_results.py`

- 扫描 `performance_result/*/closetest_result_*.txt`，每个任务取最新文件
- 输出 `performance_result/summary_YYYYMMDD_HHMMSS.txt`：
  - 每条轨迹的详细数据（task, correct_space, SR, collision, SPL, APE, AOE）
  - 全量汇总（平均值 ± 标准差）：车位选择成功率、SR、碰撞率、Mean SPL、Mean APE、Mean AOE

## 11. 关键设计决策

| 决策 | 原因 |
|---|---|
| 图片不做客户端 resize | 服务端 `PrismaticImageProcessor` 统一处理，保证与训练时预处理一致 |
| VLM yaw 单位为度 | 服务端 API 规范要求 |
| 内部全部在 camera_init 系 | 避免频繁 TF 查询，只在 publish 时转换 |
| 探索阶段丢弃 VLM yaw | VLM yaw 输出不连续，Pure Pursuit 从几何关系自行推导转向，无需 VLM yaw |
| 探索阶段用 Ackermann 连续控制 | `set_transform` 会产生跳跃，破坏 FastLIO 构建的栅格地图 |
| 只跟踪 VLM 第 4 个路点 | 提供足够前瞻距离，避免频繁重规划，同时不超出感知范围 |
| 泊车入库仍用 set_transform | Hybrid A* 路径精确，停止阶段不需要栅格地图，跳跃代价可接受 |
| 泊车前先静默 Ackermann | 防止两个执行器同时发指令产生冲突（方案 B） |
| VLM 推理时间作为自然减速缓冲 | VLM 推理 ~5–10s，车辆在此期间以 `speed=0` 等待，无需额外刹车逻辑 |
| 轨迹 Marker 固定 id=0 | RViz 以 ns+id 唯一标识 Marker，固定 id 使新轨迹替换旧轨迹，始终只显示最新一条 |
| FastLIO 每任务重启 | camera_init 必须以新起始点为原点，否则累积漂移污染下一个任务 |
| init_ego 在 FastLIO 之前 | 防止 LiDAR 跳变破坏里程计 |
| AOE 取 `min(diff, \|180-diff\|)` | 消除头朝里/头朝外的 180° 歧义 |
| rosrun 阻塞作为任务完成信号 | `rospy.signal_shutdown()` → `rosrun` 退出 → batch 脚本继续下一任务 |
| 结果存到功能包内 `performance_result/` | 与任务数据集分离，便于管理 |
